# 项目介绍话术 — 面试用

> **使用场景：** 面试官说"介绍一下这个项目"，你按下面的脚本讲 2-3 分钟。  
> **讲的时候：** 可以边讲边在白板上画架构图（文末有图），指着图讲比纯口述清晰得多。

---

## 一、30 秒电梯演讲（如果面试官只给你半分钟）

> "我做了一个基于 LangGraph 的自主推理医疗诊断 Agent。核心思路是用知识图谱约束大模型的输出，解决医疗场景的幻觉问题。架构上用了 ReAct 循环——先做实体识别、然后 LLM 决定调什么工具、查到知识图谱数据后再综合回答。特别做了双 Agent 校验机制——一个 Agent 负责诊断，另一个独立的 Agent 负责审核，发现风险不直接拦截，而是标注出来。另外做了四级记忆体系，包括用 FAISS 向量检索历史相似病例来辅助诊断。从 Streamlit 原型一直迭代到了 FastAPI 生产级服务。"

---

## 二、完整介绍脚本（2-3 分钟）

按下面顺序讲，每条展开一句话即可。面试官追问再展开细节。

### 第 1 步：一句话定调（10 秒）

> "这个项目是把一个传统的医疗知识图谱问答系统，改造成了能自主推理、自主调用工具、自主校验的 LangGraph Agent。"

### 第 2 步：为什么选 LangGraph 而不是 LangChain AgentExecutor（15 秒）

> "医疗场景和通用 ChatBot 不一样——回答错了可能误导患者。LangChain 原生的 AgentExecutor 是个黑盒，LLM 什么时候调了工具、工具返回了什么、循环了几次，你都不知道。LangGraph 用有向图把每一步显式定义出来——NER 抽取、LLM 推理、工具执行、安全校验——每个节点是独立函数，出问题能定位到具体哪一步。"

### 第 3 步：边画图边讲核心架构（40 秒）⭐ 最重要

> "我画一下核心流程。用户输入进来，先走 preprocess 节点做 NER 实体抽取。这里我用了三合一方案——RoBERTa 深度模型做语义理解、AC 自动机做医学词典快速匹配、TF-IDF 做实体名归一化对齐，三个并行跑然后合并去重。
>
> 实体抽取完进入 llm_planner 节点，这是诊断 Agent，用的 qwen-max。LLM 拿到实体先验后自己决定——需要查知识图谱就生成 tool_calls，查完回来再基于真实数据回答。
>
> 这里有两条分支：如果 LLM 出了 tool_calls，走 tool_executor 节点执行 Neo4j 查询，执行完回到 LLM 继续推理——这就是 ReAct 循环。如果 LLM 出了最终回答，走 reflection 节点做安全校验。
>
> reflection 是双 Agent 协同的关键：规则护栏检查违禁词和免责声明，再起一个独立的校验 Agent，用 qwen-plus，审核诊断 Agent 的回答——药品名是不是都在知识图谱里、检查项目有没有编造、事实是否一致。发现问题不拦截，在原文顶部加风险横幅提醒用户，底部附审核报告。"

### 第 4 步：讲两个最亮眼的点（30 秒）

> "两个我觉得比较有意思的设计：
>
> 一个是记忆体系分了四层。感知记忆存 NER 实体，工作记忆缓存当轮查询结果避免重复查，向量记忆用 FAISS 加 BGE 嵌入做历史病例的语义检索——用户问'头痛难受该吃啥药'能自动匹配到之前'头痛发烧吃什么药'的病例，相似度 74.7%。长对话超过五轮自动把旧消息压成实体摘要，Token 省了约 60%。
>
> 另一个是容错设计。工具调用有三层防护——指数退避重试、返回结果 Schema 校验、Neo4j 挂了降级到静态医学字典。LLM 这边有八级降级链——qwen-max 不行切 qwen-plus，再不行切 qwen3-vl 系列，最后还有本地 Ollama 兜底。"

### 第 5 步：讲工程演进（20 秒）

> "工程上也做了迭代。最初是 Streamlit 单体应用，后来把 Agent 核心抽出来，支持 asyncio 原生协程处理高并发。CPU 密集的 NER 推理用了多进程 Worker 池绕过 GIL。Neo4j 查询层拆成了独立的 FastAPI 微服务，方便独立扩缩容。最后写了 24 个测试用例覆盖单元到端到端。"

## 三、面试官追问的应对策略

面试官追问哪个点，你就展开哪个。下面按简历四个 bullet 组织。

### 追问 Bullet 1（架构）

**"你刚才说的 conditional_edges 是怎么控制路由的？"**

> "conditional_edges 是 LangGraph 的条件边机制。我在 AgentState 里定义了一个 next_action 字段，LLM 推理完后会设置它——如果 LLM 返回了 tool_calls，next_action 设成 continue，条件边就路由到 tool_executor；如果没有 tool_calls，next_action 设成 end，条件边就路由到 reflection 做安全校验。tool_executor 执行完也有一个 should_loop_back 判断，如果需要继续推理就回到 LLM，否则结束。"

**"校验 Agent 和诊断 Agent 用不同模型，为什么？"**

> "不能让一个模型自己审自己。诊断用 qwen-max 是因为它推理能力强、tool calling 成功率 95% 以上。校验用 qwen-plus 是因为审核任务不需要最强的模型，而且它是独立实例，不会和诊断 Agent 有系统性偏差。这就像医院里开药的医生和审方的药师是两个人。"

### 追问 Bullet 2（NER + 工具链）

**"三合一 NER 的三个组件是怎么协作的？"**

> "三个组件是并行运行的，各自从原始文本出发，最后把结果合并去重。RoBERTa 做深度语义理解，但罕见药名它可能漏；AC 自动机用医学词典做模式匹配，词典里有的一定能匹配到；TF-IDF 做最后归一化——比如用户输入'布洛芬缓释胶囊'，知识图谱里只存了标准名'布洛芬'，TF-IDF 用字符级 n-gram 算相似度，把'布洛芬缓释胶囊'对齐到'布洛芬'，不然下游查不到。"

**"工具调用的成功率从 85% 到 98% 怎么做到的？"**

> "统计了大概 50 次工具调用的日志。85% 阶段只有基础 try/except，失败主要两个原因：Neo4j 偶发抖动连接超时，和 LLM 偶尔把参数名写错。我加了三层：指数退避重试解决临时抖动，JSON Schema 校验在工具返回后检查结构是否完整，静态字典降级在 Neo4j 彻底不可用时用预置医学数据兜底。"

### 追问 Bullet 3（记忆体系）

**"四种记忆分别解决什么问题？"**

> "感知记忆是 NER 刚提取的实体，给 LLM 做先验提示。工作记忆缓存当轮的查询结果，避免 LLM 反复查同一个东西。向量记忆用 FAISS 存历史病例，新问题来时语义检索最相似的 past case 注入上下文。持久化记忆把完整对话存磁盘 JSON，关掉浏览器再打开还能恢复。"

**"为什么不用 OpenAI Embedding 而用 BGE？"**

> "一个是成本——OpenAI Embedding 按 token 计费，病例存多了费用会涨。BGE 是开源模型，本地跑免费。另一个是随手选择——BGE 在 MTEB 中文检索榜单排名前列，512 维也够用，模型只有 100MB，部署很轻量。"

### 追问 Bullet 4（高并发）

**"asyncio 和多进程 Worker 分别解决什么问题？"**

> "Agent 的大部分时间在等 I/O——等 LLM 返回、等 Neo4j 查询。asyncio 的事件循环在等 I/O 时可以切到其他协程干活，单线程就能支撑几百并发。但 NER 推理是纯 CPU 计算，asyncio 的单线程遇到 CPU 密集任务会卡住所有协程。所以我又加了多进程 Worker 池，CPU 核数乘 2 个进程，每个进程独立跑 NER 推理，绕过 GIL。2 并发实测从串行 42 秒压到 21 秒。"

**"为什么把 Neo4j 拆成独立微服务？"**

> "主要是三个考虑：故障隔离——Neo4j 挂了 Agent 不崩，自动降级到静态字典；独立扩缩容——如果查询量大了，可以单独给工具服务加实例；技术栈解耦——以后工具层可以用 Go 重写，Agent 核心不用改，保持 HTTP 接口一致就行。"

## 四、白板架构图（面试时画出来讲）

```
┌──────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────┐
│preprocess│───→│ llm_planner  │←──→│tool_executor │    │reflection│
│(ONNX NER)│    │(诊断Agent)   │    │(Neo4j+降级)  │───→│(安全校验)│──→ END
│三合一     │    │qwen-max     │    │三层容错      │    │护栏+     │
│提取实体   │    │+4个工具绑定  │    │              │    │校验Agent │
└──────────┘    └──────┬───────┘    └──────────────┘    │qwen-plus │
                       │                                └──────────┘
                有tool_calls?───是──→ tool_executor
                       │                    │
                      否                    ▼
                       │            继续推理?──是──→ llm_planner
                       ▼                    │
                  reflection              否 → END
                       │
                      END

记忆层:
  感知记忆(raw_entities) → 工作记忆(knowledge_cache) → 向量记忆(FAISS+BGE) → 持久化(JSON)
```

**讲的时候用手指着节点顺序：**
1. 指 preprocess → "输入先进这里做实体抽取"
2. 指 llm_planner → "然后 LLM 决定下一步"
3. 指条件分支 → "有工具调用走左边，没工具调用走右边校验"
4. 指循环箭头 → "工具执行完回到 LLM 继续推理，这就是 ReAct 循环"
5. 指 reflection → "最终回答必须过这一关才能返回给用户"

## 五、如果面试官说"再展开讲讲"——选一个深讲

面试官如果让你详细展开，挑一个你最熟悉的点深入讲 1-2 分钟。推荐选 **双 Agent 协同校验** 或 **三合一 NER**，因为这两个点最体现技术思考。

### 深讲双 Agent 协同校验

> "双 Agent 的核心思想是'不让一个人既当裁判又当运动员'。诊断 Agent 用 qwen-max 生成回答后，不是直接返回，而是先经过 reflection 节点。
>
> 第一步是规则护栏——纯 Python 规则，不调 LLM。检查 8 条违禁词比如'保证治愈''100%有效'，如果出现了就过滤掉。还有免责声明检查——如果 LLM 忘了加'仅供参考'，自动补上。
>
> 第二步是 LLM 事实校验——这是独立的校验 Agent，用 qwen-plus。我给它的 prompt 是：'你是医疗回答审核员，下面是知识图谱查到的事实，下面是诊断 Agent 的回答，请逐一检查药品是否编造、检查项目是否编造、事实是否一致、是否遗漏重要信息'。它输出结构化的审核结果。
>
> 这里有个设计取舍——发现问题后要不要拦截？我选的是不拦截，而是标注风险。因为如果 LLM 的回答 80% 是对的、20% 有问题，完全拦截掉用户啥也看不到，体验很差。我的做法是在回答顶部加醒目的风险横幅，比如'⚠️ 回答中的药品推荐可能缺乏知识图谱支撑'，然后在底部折叠审核详情。这样就兼顾了安全和体验。"
